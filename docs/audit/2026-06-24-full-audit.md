# Pulse full-audit — 2026-06-24

Multi-agent audit (~22 finders × adversarial verify × synthesis). **70 raised → 55 confirmed.**

Severity: critical 7, high 14, medium 27, low 7.

## Synthesis

29 confirmed findings collapse into ~9 themes. The genuinely critical, ship-now set is small but severe: (1) an unauthenticated PandaDoc webhook that lets anyone on the internet forge contract-signed events and overwrite money/contract dates; (2) a deactivated employee retaining full PII read access via direct PostgREST; (3) marketing campaigns that bypass the suppression list and email customers/partners/opted-out contacts (CAN-SPAM exposure for a healthcare-compliance vendor); and (4) two inline opportunity-discount paths that silently zero or go NEGATIVE on amount-only imported deals — the exact stale-overwrite money-corruption class this audit was chartered to hunt. A second tier of high-severity correctness bugs (unauthenticated ARR view leak, forecast filtering on dead enum values, zeroed churn/NRR KPIs, renewal every-other-year no-op, task-reminder duplicate-email bursts, renewals-queue date bucketing dropping money) is user-visible-wrong but not actively destroying data. The rest are medium/low fragility and hygiene. Note: several findings are cross-dimension duplicates (PandaDoc signature ×3, suppression ×7, task-reminder recurrence ×4, renewal lookback ×2, RQ invalidation ×3) and are merged below.

### Recommended fix order

- 1. PandaDoc webhook HMAC verification + reject-on-fail (unauthenticated contract/money write — highest blast radius, easiest to weaponize). Same change kills the duplicate integrations-health findings.
- 2. Lock down reads for deactivated users: AND current_app_role() into core SELECT policies + revoke session on deactivate (PII/offboarding leak). Then correct the false ProtectedRoute security comment.
- 3. Revoke anon from v_dashboard_arr_financial (one-line migration, financial-data leak).
- 4. Enforce marketing suppression server-side in playbook-smartlead (subtract v_marketing_suppression for ALL sources), and tighten the partner-reason over-suppression. Closes the entire CAN-SPAM cluster (7 findings) at the one authoritative chokepoint.
- 5. Fix the two inline opportunity-discount money bugs in OpportunityDetail (zero-on-import + missing negative floor) — same edit; mirror OpportunityForm's 0-subtotal guard. While here, fix the DB skip-no-lines discount trigger so non-UI write paths don't drift.
- 6. Forecast dead-enum filters + Team Dashboard lifecycle_status-derived KPIs + Churn% double-counting denominator — the leadership-facing numbers that are silently wrong.
- 7. Renewal automation correctness batch: every_other_year parity, add lookback_days window (both migrations), gate on derived deal reality not manual status, leap-year term-length guard, per-row savepoint isolation in generate_upcoming_renewals.
- 8. task-reminders recurrence/dedup/silent-advance + task-digest idempotency & overdue cap (reminder spam + double digests).
- 9. react-query invalidation cleanup: fix useAddOpportunityProduct's missing detail invalidation, delete the dead singular ["opportunity"] key everywhere (key factory), and add broad ["opportunities"]/["pipeline"] invalidation to line-item mutations so pipeline/ARR refresh.
- 10. ### Themes

- **Unauthenticated / under-authorized write & read holes (security/compliance)** — PandaDoc webhook (no-op signature + --no-verify-jwt) is an open contract/money write; deactivated users keep PII read access via PostgREST; v_dashboard_arr_financial leaks financials to anon; recalc_opportunity_amount granted to anon; leftover world-writable public.test table; user_profiles self-update denylist. These are the true criticals plus latent escalation surface — fix the writes and PII reads first.
- **Marketing suppression not enforced at send time (CAN-SPAM)** — 7 merged findings: v_marketing_suppression is only a report; the actual send path (fetchRecipientsByTag + CSV + paste → playbook-smartlead) emails customers, partners, account-DNC and opted-out contacts. Compounded by the partner-reason OVER-suppressing real prospects. One server-side chokepoint in the edge function fixes the whole class in both directions.
- **Opportunity money math — stale/computed overwrite class** — The chartered bug class, found in 4 more places: inline discount zeroes amount-only deals (NULL subtotal), inline $-discount writes negative amounts (missing floor), the DB skip-no-lines discount trigger freezes stale amount for non-UI paths, and the renewal generator's parent-amount copy is silently overwritten by the line-item recalc trigger.
- **Reporting / financial correctness shown to leadership** — Forecast 'Best Case' filters on dead enum values (always $0); Team Dashboard NRR/churn/starting-customers key off the uniformly-'prospect' lifecycle_status (read 0); churn% denominator double-counts won+lost accounts; renewals-queue month tiles & range filter drop/shift money via UTC-vs-local date bucketing; ARPC undercounts multi-year contracts; ReportBuilder charts plot per-row instead of aggregating; raw-export silent 50k truncation.
- **Renewal automation robustness & accuracy** — every_other_year skip is a no-op for annual contracts; no lookback window (past-due anniversaries skipped forever, reported in two migrations); gates on manually-set accounts.status that drifts instead of derived deal reality; leap-year guard ignores multi-year term length; whole-batch abort on one bad row (no per-row savepoint).
- **Reminders & digests — duplicate/spam and silent loss** — task-reminders re-fires recurring reminders one-per-tick through a backlog, has no dedup guard, and advances the schedule even when the email send failed (lost occurrence); task-digest has no per-day idempotency (re-run double-emails) and an uncapped overdue section.
- **react-query cache invalidation drift** — Dead singular ["opportunity"] key in 5 mutations reads by zero queries; useAddOpportunityProduct never invalidates the real detail key so added-product Amount stays stale; line-item mutations never invalidate list/pipeline/ARR keys so those surfaces show pre-edit Amounts. A shared key factory prevents recurrence.
- **Silent failures masked as success/empty** — Bulk owner-change swallows per-row PostgREST errors (false success toast); core list pages render the empty-state on query failure instead of an error/retry; task-reminder schedule advances on failed send.
- **Integration & data-hygiene fragility** — PandaDoc activity insert non-idempotent on webhook retries + matches to an arbitrary opportunity on fuzzy fallback; account merge doesn't reparent campaign_enrollments.account_id; sync-emails inner loop unbounded vs 150s gateway; Smartlead never sets stop-on-reply; referring_partner imported into an invisible column; contacts 'Account' sort is a no-op; v_partner_accounts DROP pattern can cascade-drop the suppression view.

## All confirmed findings

### CRITICAL

- **Deactivated user can still read all account/contact/opportunity PII via PostgREST (RLS read policies never check is_active; session not revoked)** _(auth-offboarding, confirmed)_
  - `supabase/migrations/20260331000000_initial_schema.sql:483-487, 506-510, 552-556`
  - Impact: The SELECT policies on the core CRM tables only require the caller to be in the `authenticated` Postgres role — they never call current_app_role() or check user_profiles.is_active (unlike the INSERT/UPDATE policies, which do gate on current_app_role()). Deactivation (UsersManager / deactivate_brayden) only flips is_active=false; it never calls auth.admin.signOut/revokeRefreshToken/deleteUser, so a departed employee keeps a valid refresh token for its full lifetime. With that token they can hit the PostgREST REST API directly (e.g. GET /rest/v1/contacts?select=*) and read every account, contact (name/email/phone), and opportunity (amounts), bypassing the ProtectedRoute client gate entirely. For a healthcare-compliance company this is a PII/compliance exposure on offboarding. The 20260514000007_read_only_role.sql comment even documents the root cause: 'SELECT policies are uniformly `to authenticated using (true)` so [any logged-in user] can still SELECT everything.'
  - Fix: Gate reads on active status too: change these SELECT policies to `using ((archived_at is null or public.is_admin()) and public.current_app_role() is not null)` (current_app_role() already returns NULL for inactive users), OR add a dedicated `public.is_active_user()` predicate and AND it into every read policy. Additionally, on deactivation call supabase.auth.admin.signOut(userId, 'global') (or admin.deleteUser for true offboarding) from the invite-user/a new admin edge function so the existing refresh token is invalidated immediately rather than relying on a column flip.

  - `src/features/playbook/CampaignRecipients.tsx (CSV/paste paths) + src/features/playbook/api.ts:636 (tag path) + supabase/functions/playbook-smartlead/index.ts:264 (launch add-leads):CampaignRecipients.tsx:71-85 (importCsv/applyPasted), csv.ts:61-84, smartlead/index.ts:264-287`
  - Impact: For a healthcare-compliance company, a rep can paste or upload a list that includes a contact flagged do_not_contact / no_longer_employed (or an unsubscribed address) and the campaign mails them. The CRM's own suppression flag is silently ignored for two of the three recipient sources. CAN-SPAM / opt-out compliance hole and a reputational/deliverability risk.   - Fix: Before launch, resolve every recipient email against contacts and drop any with do_not_contact/no_longer_employed/archived (or an unsubscribe flag), regardless of source. - **PandaDoc webhook signature verification is a no-op — public endpoint lets anyone forge contract signings and overwrite contract dates** _(edge-functions, confirmed)_
  - `supabase/functions/pandadoc-sync/index.ts:71-91, 418`
  - Impact: The function is deployed with --no-verify-jwt (no JWT auth) and the ONLY gate is this signature check, which returns true on every code path — missing header, mismatch, or match. Anyone who knows/guesses the function URL can POST a crafted {event:'document_state_changed', data:{status:'document.completed', recipients:[{email: <any real contact>}], name:'...', date_completed:'...'}} body and the function will: match a real contact by email, overwrite that opportunity's contract_start_date/contract_end_date and the account's current_contract_start_date/current_contract_end_date, and insert a fake 'Contract signed' activity. That is unauthenticated write access to money/contract data and forged audit history. The _body param is even named '_body' (never hashed). The real HMAC-SHA256 verification the comment promises is never implemented.
  - Fix: Implement real verification: compute HMAC-SHA256 of the raw request body using the PandaDoc shared key (not the API key) and compare (constant-time) against the X-PandaDoc-Signature header; reject (401) on missing/invalid signature instead of returning true. Do not 'allow through during setup' in production. Until implemented, treat this endpoint as an open write hole.
- **PandaDoc webhook signature verification is a no-op — public endpoint lets anyone forge contract-signed events and overwrite opportunity/account contract dates** _(integrations-health, confirmed)_
  - `supabase/functions/pandadoc-sync/index.ts:71-91 (verifyWebhookSignature); deployed --no-verify-jwt per line 12`

- **Inline discount edit zeroes Amount on amount-only (productless, imported) opportunities** _(opp-money-math, confirmed)_
  - `src/features/opportunities/OpportunityDetail.tsx:701-713`
  - Impact: For productless / amount-only opportunities (the majority of SF imports, where deal value lives in opportunities.amount and there are NO line items), opportunities.subtotal is never written by recalc_opportunity_amount (it bails on 0 line items, see 20260426000006 / 20260430000003), so it stays NULL. - **Bulk "change owner" silently swallows per-row failures (false success)** _(pagination-dedup, confirmed)_

- **every_other_year skip never fires for annual contracts (cycle_count is NULL) — biennial accounts get renewed every year** _(renewal-automation, confirmed)_
